一、估价逻辑：先分清「赛道」，再谈数字

企业为「头寸可见、预测可解释、计划可排程」付费时，很少只买一个 SKU，而是买能力组合：流水与主数据管道、预测与情景模型、计划偏差与预警、可选的 DataAgent 问数层。不同赛道定价锚点不同，混在一桌比价会失真。

1）订阅 / 许可锚点

国际 SaaS 常以 年费 + 模块 + 交易量/实体数 计价；国内厂商常以 平台许可 + 实施人天 为主，年费维保多为合同额的固定比例。

2）实施成本锚点

采购与支出管理类套件在买方市场报告中常见实施费 = 首年订阅的 50%–150%（视集成深度而定）。TMS/司库类项目则高度依赖银企与 ERP 接口数量。

3）AI 增量锚点

大模型调用通常按 Token / 席位 / 私有部署算力 单列；Agent（多轮 + 工具 JSON）会把单次问答的 token 显著放大。公开单价见下文 第五节，可与首年软件费分项列示。

4）本报告数字怎么用

下表「价格带」多为公开市场二手归纳或量级引用，用于框定谈判心理区间；正式投标以厂商报价与贵司法务为准。

二、竞品地图（按市场逻辑分组）

「竞品」对资金预测智能体而言分三类：替代型（客户可能只买 TMS/司库套件不买单点）、互补型（已有银企与支付执行，需要预测与计划专题）、挤压型（IT 自建 Copilot）。下表列出主流玩家与典型卖点，便于销售话术对齐。

分组 代表产品 / 厂商 核心能力与客户心智 与亿流关系
国际 TMS Kyriba、SAP Treasury & Risk、Oracle Treasury、FIS 等 全球银行连接、流动性、支付工厂、市场风险；大单长周期，按模块与实体规模扩容。 互补为主：亿流不替代银企与支付工厂，专注预测、计划排程、偏差预警与 DataAgent 问数闭环。
P2P / 支出 Coupa、SAP Ariba、Jaggaer（视区域）等 采购、供应商、合同到发票；流程广度大，现金流侧常需与应付计划手工对表。 部分替代张力：若客户已上全套，可能只需轻量扩展；若仅上部分模块，亿流可补资金预测与计划专题。
AP 自动化 HighRadius、Basware、Medius 等 发票匹配、应付自动化、部分带 AI Agent；计价常与发票量挂钩。 场景相邻：亿流更强调头寸与预测口径、计划偏差闭环，而非单纯发票处理自动化。
国内司库 用友 BIP 司库、金蝶星瀚/云·司库、九恒星、拜特、中兴新云 等 账户可见、资金池、结算、预算衔接；常随集团司库体系建设打包招标。 强互补：亿流可作为现金流预测与资金计划专题或 AI 能力子项目嵌入场内。
实施与集成 汉得、赛意、各大四大咨询分包等 人天单价清晰，交付以里程碑验收；软件边界有时模糊。 渠道/共建：亿流提供产品壳与 Agent 参考实现，集成商铺接口与测试。
FP&A / EPM Anaplan、Workday Adaptive、Oracle EPM 等 预算、滚动预测、业务动因；常与司库分线采购，需对齐现金口径与时间桶。 互补或并存：亿流侧重头寸、流水与支付侧预测；与 FP&A 联动时以接口与主数据合同化。
流水·预测专题 见知 Xencio 等（公开产品页常谈多方法预测） 垂直「现金流预测」叙事；多为询价 + 实施，适合作为 POC 对标对象。 场景相邻：亿流强调与司库工作台同壳、假设版本、DataAgent 问数与导出的可验收组合。
泛 AI Azure / Copilot 生态、企业微信 / 钉钉 侧 GPT 应用、各类 RAG 外包 灵活、起步快；缺垂直 KPI 与审计叙事时需长期定制。 挤压型：客户若只愿买「一个对话框」，需用闭环产品与审计说服升级。

国际 TMS 公开叙事参考：Kyriba 等在官网用例中描述 cash forecasting、流动性与场景分析能力（如 Cash Forecasting、 2026 Cash Forecasting Essentials）。 页面中的准确率类表述多为营销或特定样本，投标须按客户数据单独验证。

四、亿流资金预测智能体：产品差异化小结

差异化应落在客户可感知的采购理由上：谁拍板、谁验收、谁运维。

市面常见组合

  • 国际套件：流程强，但落地与二开成本高，AI 多为增量模块。
  • 国内司库：银企与结算强，预测模型、情景与计划偏差有时仍靠 Excel 与会议。
  • 纯 AI 项目：演示快，缺口径版本、偏差日志、运维看板时难过年审。

亿流方案

  • 垂直闭环：总览、资金流、分析预测、资金计划、集成运维——同一产品叙事。
  • DataAgent 嵌入分析侧栏：工具调用对齐真实数据路径，适合作为问数与安全评估 POC。
  • 先对齐口径再加深模型：先「看得见头寸与计划」再「可解释预测」，贴合司库与 FP&A 节奏。
  • 与司库一体化工作台同壳：可演进为 iframe 能力模块，降低培训成本。

5.3 Agent 为什么比「单次问答」费 token

本工程 DataAgent 使用 Function Calling：常见路径为「用户提问 → 模型输出 tool_calls → 后端执行数据库工具 → 结果作为 tool 消息写回 → 模型再生成自然语言答复」。因此至少 2 次对模型的可见请求，且每轮重复发送 system 与 tools JSON 模式。经验上，同一用户问题最终可计费 tokens 约为「单次纯聊天」的 约 2～5 倍。下文公式用单次有效会话的合并 tokens（已含多轮）估算。

5.4 模型 API 年费 · 精确计算逻辑（六步）

以下符号与步骤可直接写入 SOW 附件「模型费测算说明」。

符号 U = 月活跃使用助手的用户数（人） q = 人均每个工作日发起「有效会话」次数（次/人/日） W = 每月工作日天数（常取 22） Tin = 单次有效会话合计输入 tokens（含多轮、system、工具定义回填） Tout = 单次有效会话合计输出 tokens Pin = 公示价：输入，元 / 百万 tokens Pout = 公示价：输出，元 / 百万 tokens ① 月有效会话数：S = U × q × W ② 月输入 tokens：Min = S × Tin ③ 月输出 tokens：Mout = S × Tout ④ 月 API 费用（元）：C月 = (Min / 106) × Pin + (Mout / 106) × Pout ⑤ 年费用（元）：C年 = C月 × 12 ⑥（可选）加风险缓冲：C′年 = C年 × (1 + β)，β 建议 0.15～0.30（峰值、活动价结束、推理模型混用）

六、亿流初步估价体系（人民币 · 首年）

采用「软件授权/订阅 + 实施 + 运维 + Token」四分法；其中 Token 的可核算口径见上一节 第五节，与统一提案幻灯 unified-proposal/index.html 保持一致，便于销售、交付、财务同一页对齐。

首年软件+实施区间怎么算（对内口径）

公司统一人天单价 ￥2,500/人天；与提案人周表对齐时取1 人周 = 5 人天。供应侧合计人周（试点 22–34、标准 50–84、集团 95–147）对应的实施费（万元）为 人周 × 5 × 0.25 = 人周 × 1.25（0.25 = 2500÷10000），即约 27.5–42.5 万、62.5–105 万、118.75–183.75 万。首年三档总价 = 上述实施 + 软件许可/订阅摊销与风险储备（随范围浮动），对外区间与下表一致。

试点 / POC

约 70–140 万

单法人或单业务域；2–3 个系统对接；标准预测与计划口径；助手以只读查询为主，写操作白名单极小。

TCO 拆分建议（对内模型）

  • 实施人天：按￥2,500/人天 × 人周表合计 × 5；变更人天单价或人周假设时，应同步重算区间并与销售、交付对齐。
  • 年费运维：建议按首年合同额 18%–22% 预估（热线、升级、规则运营顾问包可拆 SKU）。
  • 模型成本：按 第五节 5.4～5.7公式与计算器复算；对外可报「每人每月固定次数包 / 年包 tokens + 封顶」避免账单抖动。
  • 集成人天：每个异构系统接口按 80–200 人天量级做风险储备（视 ERP 年代与主数据质量）。
  • 验收锚点：预测偏差 KPI、计划闭环时效、导出字段清单、助手动作白名单与安全渗透结论。
