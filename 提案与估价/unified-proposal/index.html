司库与财务现场常听到的抱怨

问题往往不在「缺一个模型」，
在流水、计划、预测各算各的。

打个比方：网银流水、业务计划、预测模型像三张表——会上问「下周钱够不够」，往往要临时拼 Excel，谁也不敢拍胸脯。

账户流水 银企/导入 + 业务计划 收支排程 → 预测与预警 可解释
  • CFO 要回答「未来 30/60 天头寸」，但预测假设散落在业务线，一调参数全员重算。
  • 在线户与离线户并存：直连能自动拉流水，离线户仍靠导出——需要统一汇聚与对账口径（纪要：账户类型决定取数方式，目标仍是「全量资金视图」）。
  • 计划场景（纪要）：例如票据账期确定时，司库要在到期日前保证头寸；账期内可安排短期运作——预测与计划必须把承诺支付日、在途与理财占用口径对齐，否则「下周钱够不够」仍说不准。
  • 行业差异 → 产品形态（纪要）：电商、大宗等业务逻辑差很大，「一个通用预测工具包打天下」不现实；交付上宜通用底座（方法论 + 计算与版本规则）+ 垂直行业 Skills/知识包（标签、假设模板、校验规则）迭代，智能体按上下文选用。
  • 先统一数据再谈算法：科目、账户、业务标签对齐后，预测与偏差预警才可解释、可复盘；对内常说「数据底座」，对客户讲一屏能讲清钱从哪来、要到哪去。
  • TMS/司库套件强项在银企与支付执行；本方案侧重预测、计划排程与偏差闭环——与风控「支付前对齐」、与 TMS 相邻互补，不承诺替代整套司库。

与「支付前内控 / 风控智能体」同属司库智能化方向；本套材料主讲现金流预测、计划与头寸可见。右侧为资金流实机（?embed=1#cashflow）；本地起前端服务即可 Live。

口语化对齐访谈；经营性现金流 + 计划编报常被选作第一个切口。客户常先问「收费模式」——接后文「三笔钱」与三档区间。

演示与 POC 建议当场说清的三点

取数、预测口径、侧栏问数——一条线验收

01

流水从哪来、多勤更新

银企直连与离线导入并存时，把接口/文件格式、对账频率（T+0 / T+1）写进合同附件；可先脱敏样例跑通再接生产。

02

预测假设谁定、怎么改

业务参数、季节性、票据账期等共建字典；模型与假设有版本、可回滚；默认提示与解释，硬拦截节奏与财务制度对齐。

03

DataAgent 边界

侧栏问数、解释偏差、带出依据；受控工具读库。建议不等于司库决策——全程留痕、可导出。

若集团同步上支付前内控 / 风控智能体，可与本方案并行：一个管「钱怎么出去」，一个管「钱怎么预见与排程」——与 TMS 执行层互补。

先数据与字段，再模型；各审批节点展示粒度可不同。

切口明确

聚焦现金流预测、计划排程与偏差闭环；不承诺替代完整 TMS/司库套件，也不包圆全集团所有业务线模型定制。

底座 + 行业包（纪要）

承认行业差异：通用方法论与计算底座稳住版本与日志；可迭代的 Skills/知识包（标签、假设模板、校验）覆盖电商/大宗等逻辑。

可 POC 验证

工程可本地跑通；预测假设、DataAgent 与导出可在尽调中逐项打开，不靠「只有 PPT」。

叙述顺序建议：痛点（口径散、预测不可信）→ 范围（预测与计划）→ 里程碑（流水—口径—模型—助手）→ 价钱与风险。

少讲模型名，多讲字段、日志与验收标准。

附录 2/9 · 一、估价逻辑

先分清赛道，再谈数字

企业采购常同时买：数据连通、流程/控制、后续运维、可选模型调用——若混在一起只比总价，容易误判结构。

订阅 / 许可国际 SaaS：年费+模块+交易量；国内：许可+人天，维保常按合同额比例。
实施成本P2P 类实施费常见 首年订阅 50%～150%；司库/TMS 看银企与 ERP 接口数。亿流交付侧对内核算￥2,500/人天（人周×5，与人周专页、首年区间一致）。
模型与算力Token、席位或私有部署算力宜与软件费分列；带工具调用的场景通常比纯对话明显多耗 token，合同里写默认模型与切换公式。
比价怎么用价格带是场景化锚点，不是一口价；正式报价以盖章报价单 + SOW与法务为准。

  • 纯 AI：演示快，缺规则版本与审计底稿难过审。

亿流

  • 一条资金叙事：总览、流水、计划、预测、预警同一套口径。
  • 分析侧栏可查库：DataAgent 走白名单工具，答复带来源，便于抽查。
  • 控制强度可分阶段；需要时可与司库/TMS 门户做 iframe 或单点并存。

标准交付
